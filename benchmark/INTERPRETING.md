# 벤치마크 해설 & 결과 해석법

이 문서는 **이 벤치마크가 무엇을 재는지**와 **결과(RESULTS.md)를 어떻게 읽어야 하는지**를 설명한다.
실행 방법은 [START-HERE.md](START-HERE.md), 재현 레퍼런스는 [REPRODUCE.md](REPRODUCE.md),
실제 수치는 [RESULTS.md](RESULTS.md)에 있다.

---

## 1. 무엇을 비교하나

**같은 코딩 과제를, 같은 모델(qwen-122b), 같은 조건으로 두 에이전트 도구(codex · openhands)에 시켜서
"도구의 차이"만 드러낸다.** 모델을 통일했기 때문에 결과 차이는 모델이 아니라 *도구의 작업 방식*에서 온다.

핵심 설계 원칙 3가지:
- **동일 조건** — 같은 프롬프트, 같은 모델(:4000 qwen-122b), 격리된 작업 폴더, 직렬 실행.
- **공정한 채점** — 성공 여부는 도구의 "다 됐다"는 자기보고가 아니라 **독립 채점기(judge)** 가 결정.
- **정직한 기록** — 실패·재측정·이상치를 숨기지 않고 RESULTS.md에 남긴다.

---

## 2. 과제 3단계 (복잡도 사다리)

| 레벨 | 과제 | 무엇을 보나 | 채점 방식 |
|------|------|-------------|-----------|
| **L1** | `fib(n)` 단일 파일 | 기본 함수 한 개를 정확히 | judge가 `fib` 를 **import** 해 값 검증 |
| **L2** | `wordstat` 멀티파일 CLI | 여러 파일로 나눈 CLI, 표준출력 계약 | judge가 **subprocess** 로 실행해 stdout/종료코드 검증 (블랙박스) |
| **L3** | KV 스토어 멀티모듈 서비스 | 패키지 구조 + 프로세스 간 영속성 | judge가 **별도 프로세스**로 CLI를 호출, 영속/종료코드 계약 검증 |

레벨이 올라갈수록 "파일 1개 함수" → "여러 파일 협력" → "모듈 패키지 + 상태 영속"으로 **요구 복잡도가
커진다.** 쉬운 과제는 둘 다 통과하기 쉽고, 어려운 과제일수록 도구의 한계(끝까지 완성하는 힘)가 드러난다.

> judge는 **숨겨져 있다** — 프롬프트(PROMPT.md)는 테스트 내용을 모른다. 그래서 한 채점기로 어떤 도구의
> 산출물이든 공정하게 판정할 수 있고, 도구가 테스트에 과적합할 수 없다.

## 3. 4가지 지표 — 의미와 측정법

각 실행(run)의 `meta.json` 에 기록되고 RESULTS.md 표의 열이 된다.

### Success (passed) — 능력
- **의미:** 산출물이 실제로 과제를 푸는가.
- **측정:** 그 레벨의 독립 judge(`tasks/<level>/test.py`)를 **산출 폴더에 대해 다시 실행** → 종료코드 0이면 PASS.
- **주의:** 도구가 transcript에 "all tests passed"라고 적어도 **무시**한다. 오직 judge 재실행 결과만 신뢰.

### Time (duration_seconds) — 속도
- **의미:** 그 과제를 푸는 데 걸린 실제 시간(벽시계).
- **측정:** 실행 시작~종료 시각 차이.
- **주의:** 같은 로컬 백엔드를 직렬로 쓰므로 도구 간 절대 비교는 공정하다. 단, **실행마다 달라진다**(아래 §5).

### Steps (steps + step_method) — 작업량/과정
- **의미:** 도구가 목표까지 몇 "스텝"을 밟았나(작업 과정의 거칠음).
- **측정:** **도구마다 단위가 다르다.**
  - codex = transcript의 `exec`(셸 실행) 블록 수
  - openhands = transcript의 "Number of agent messages" 값
- **⚠️ 절대 주의:** codex의 `2`와 openhands의 `4`는 **다른 종류의 사건**이다. **raw 숫자를 직접 비교하지 말 것.**
  그래서 표에 항상 `step_method`를 같이 적는다. 같은 도구의 레벨 간 추세(L1<L2<L3)나, 같은 도구의
  실행 간 비교에만 쓴다.

### Size (files / loc) — 산출 규모
- **의미:** 도구가 만들어 낸 파일 수와 코드 줄 수.
- **측정:** 산출 폴더의 파일 수 + LOC 합. 단, `transcript.log`·`meta.json`·`__pycache__`·`.pyc`는 제외(하니스 부산물).
- **주의:** 규모가 크다고 더 좋은 것은 아니다. **PASS와 함께** 봐야 의미 있다(0 files면 보통 실패).

## 4. RESULTS.md 표를 읽는 법

표 한 행 = 한 셀(한 도구 × 한 레벨):
```
| Tool | Level | Success | Time | Steps (step_method) | Size (files / loc) |
```

읽는 순서 추천:
1. **Success 열 먼저** — 누가 어떤 레벨까지 통과했나(능력의 1차 지표).
2. **레벨별로 두 도구를 비교** — 같은 레벨에서 codex vs openhands의 성공·시간·규모.
3. **Time** — 같은 PASS라도 누가 더 빨랐나.
4. **Steps는 step_method 확인 후** 같은 도구 안에서만(또는 추세로만).
5. **0 files + FAIL** 조합이 보이면 §6(이상치 진단)으로.

올바른 결론 vs 위험한 결론:
- ✅ "openhands가 L3를 통과했고 codex는 실패했다(이 실행에서)" — Success 기반, 타당.
- ✅ "codex가 L1/L2에서 더 빨랐다" — Time 기반, 타당.
- ❌ "codex가 steps가 적으니 더 효율적이다" — **단위가 달라 비교 불가**. step_method 무시한 오독.
- ❌ "openhands가 loc가 많으니 더 낫다" — 규모≠품질. PASS와 분리하면 의미 없음.

## 5. 왜 결과가 매번 다른가 (비결정성)

LLM은 같은 프롬프트라도 실행마다 다른 출력을 낸다. 따라서:
- **한 번의 매트릭스는 "스냅샷"** 이다. 단정적 순위표가 아니라 그 실행에서 일어난 일이다.
- 특히 경계선 과제(L3)는 **같은 도구가 어떤 날은 통과, 어떤 날은 실패**할 수 있다.
- 더 견고한 결론을 원하면 같은 셀을 여러 번 돌려 통과율/시간 분포로 봐야 한다(현재 v1은 1회 측정 — v2 후보).
- `benchmark/.runs/` 는 git에 안 올라가므로 **RESULTS.md 가 그 실행의 유일한 영구 기록**이다.

## 6. 이상치 진단 — "실패"가 진짜 실패인가, 하니스 문제인가

`FAIL` 이라고 다 같은 실패가 아니다. transcript와 산출 폴더를 보고 구분한다:

| 패턴 | 가능한 원인 | 확인법 |
|------|-------------|--------|
| FAIL + **0 files**, 매우 짧은 시간 | 도구가 **일찍 잘림/포기**(진짜 실패) | transcript 끝이 중간에 끊겼나 (예: v1.0 codex L3 — 14초, `mkdir`만 하고 truncate) |
| FAIL + **0 files**, 정상 시간 | 산출물이 **다른 위치로 샘**(하니스/격리 문제) | tasks/ 등 엉뚱한 곳에 파일이 생겼나 (v1.0 openhands `--file` 누출 사례) |
| FAIL + 파일 있음 | 코드는 썼으나 **계약 위반**(진짜 실패) | judge를 직접 돌려 어떤 검사가 깨지나 보기 |

> v1.0 실제 사례: openhands L2/L3가 처음엔 "0 files FAIL"로 찍혔지만 진단해 보니 **격리 누출**(출력이
> 정규 폴더로 샘)이었다 → 러너 수정 후 재측정하니 PASS. codex L3는 **진짜 truncation 실패**라 그대로 둠.
> 이 구분을 RESULTS.md "정직성 노트"에 남긴다. **0 files FAIL은 항상 한 번 의심**할 것.
