# 8. 결과 요약과 결론

앞 장의 해석법을 **실제 14셀 측정값**(2 도구 × 7 레벨, qwen-122b)에 적용해 보자. **이건 한 번의
스냅샷**이며, 당신이 직접 돌리면 숫자는 달라질 수 있다.

## ⏩ 결과 한눈 요약 (TL;DR)

| | 통과 | 실패 | 속도 | 한 줄 |
|---|---|---|---|---|
| **codex** | L1·L2·L4·L5·L6 (5/7) | **L3·L7** | 통과 셀에서 대체로 **더 빠름** | 빠르고 폭넓게 통과, 단 **패키지 구조 과제(L3·L7)** 는 `mkdir` 직후 루프가 끊겨 실패 |
| **openhands** | **L1~L7 전부 (7/7)** | 없음 | 더 느림(약 1.3~2배) | 느리지만 7종 모두 완성 |

**어디서 갈렸나:** 분기점이 **두 곳(L3·L7)** 으로 또렷해졌다 — 둘 다 **패키지 하위 디렉터리**가 필요한
과제다. codex의 첫 행동은 빈 출력 `mkdir -p <pkg>`인데, qwen-122b는 그 다음 턴에서 도구 호출 없이
서두를 반복하고 멈춘다. `codex exec`는 도구 호출 없는 턴을 "완료"로 보고 정상 종료(exit 0)해 **파일
0개**로 끝난다. 반면 평면 파일을 바로 쓰는 알고리즘(L4)·상태(L5)·데이터(L6)는 codex도 문제없이 통과.

> **핵심 패턴 (검증됨):** codex의 약점은 "난이도"도 "토큰 한도"도 아니라 **특수한 도구 결과(빈 출력 ·
> 정책 거부)에서 에이전트 루프를 유지하지 못하는 모델 측 결함**이다. L1도 같은 ~49k 토큰에서 멀쩡히
> 통과했으니 49k는 천장이 아니다. **검증:** L7을 동일 contract의 평면(패키지 없음·`mkdir` 없음) 변형으로
> 재실행하자 codex는 13스텝을 돌려 **완전히 동작하는 솔루션**(서버·영속성·재시작·전 라우트)을 생성했다 —
> HTTP 서비스 로직 자체는 문제가 아니었다. 이 평면 실행은 **두 번째 degeneration 모드**도 드러냈다:
> 솔루션 완성 후 정리용 `rm`이 샌드박스에 거부되자 동일 명령을 **242회** 재시도하다 타임아웃.
> mkdir-stop 재현 3회(L3 2 + L7 1) + rejected-retry 1회.

**기대효과 vs 실제:**
- L1 기본기 / L2 구조화 CLI → ✅ 둘 다 통과(예상대로).
- L3·L7 "서비스 ceiling 시험" → ✅ 예상대로 갈림(codex 실패 / openhands 통과). **L7 추가로 분기 확증.**
- L4 알고리즘·L5 상태·L6 데이터 → 둘 다 통과(이 난이도에선 변별 안 됨 — 둘 다 능력 있음).

## 해석 (§7의 순서대로 읽기)

**1) Success:** codex 5/7 (L3·L7 실패), openhands 7/7.
**2) 레벨별:** L1·L2·L4·L5·L6은 둘 다 통과 → 능력 대등. **L3·L7(큰 서비스)에서 분기**(codex 실패).
**3) Time:** 통과한 셀들에서 codex가 대체로 더 빠르다(L1 23 vs 47, L4 88 vs 120, L6 142 vs 173).
**4) Steps:** 도구 내부 추세로만 — 단위가 달라 raw 비교 금지(§7).
**5) 이상치 — codex L3·L7:** 둘 다 `0 files` + 매우 짧은 시간 → `mkdir` 직후 루프 조기 종료(빈 패키지만
만들고 끊김). 토큰 천장 아님(L1도 같은 ~49k에서 통과). 격리 누출 아님(tasks/ 정규 유지 확인). **3회 재현 +
평면 변형으로 원인 검증됨.**

## 이 스냅샷이 말하는 것 / 말하지 못하는 것

**말할 수 있는 것:**
- codex는 **7종 중 5종을 빠르게 통과**하되 **패키지 구조 과제(L3·L7)에서 `mkdir` 직후 루프가 끊겨** 실패(난이도·토큰 한도 아님 — 평면 변형으로 검증).
- openhands는 **7종 전부 완성**(더 느림).
- **L7 추가가 효과적이었다** — L3 한 곳뿐이던 분기점이 "서비스 유형 = codex의 약점"이라는 **패턴**으로 확증됐다.

**말하지 못하는 것:**
- "openhands가 항상 낫다" — ❌. 통과 셀에선 codex가 더 빨랐다.
- "codex는 서비스 과제를 영영 못 한다" — ⚠️ 반복 재현됐지만 1구성(이 모델·이 설정)의 결과. 통과율로 확정 필요.
- steps로 효율 비교 — ❌. 단위가 다르다.
- "L4~L6은 변별력이 없다" — ❌. 이 난이도/모델에선 안 갈렸을 뿐.

## 결론 (균형 잡힌 읽기)

> **속도가 중요하고 과제가 단순~중간이면 codex가 유리**할 수 있고, **복잡한 멀티모듈 과제를 끝까지
> 완성하는 안정성이 중요하면 openhands가 유리**할 수 있다 — *이 한 번의 측정 기준으로는.* 확정하려면
> 같은 매트릭스를 여러 번 돌려 통과율·시간 분포를 봐야 한다.

당신의 결과로 직접 이 해석을 해보는 것이 이 책의 목적이다. 막히면 → [9. 막혔을 때](09-troubleshooting.md).
