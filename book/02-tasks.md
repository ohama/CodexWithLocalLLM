# 2. 벤치마크 과제 6종 — 선정 이유와 기대효과

비교 **대상**은 도구 2개(codex · openhands)이고, 그 둘에게 **똑같이 시키는 과제가 6종**이다(모두 CLI).
왜 이 과제들인가? 그리고 각 과제에서 무엇을 보려고 하는가(기대효과)?

## 한눈에

| 레벨 | 과제 | 복잡도 차원 | 자극하는 능력(분별 포인트) |
|------|------|-------------|----------------------------|
| **L1** | `fib(n)` 단일 파일 | single-file | 기본기 — 함수 하나를 정확히 |
| **L2** | `wordstat` 멀티파일 CLI | multi-file CLI | 파일 협력 + 표준출력 계약 |
| **L3** | KV 스토어 멀티모듈 서비스 | multi-module service | 패키지 + 상태 영속(프로세스 간) |
| **L4** | `calc` 산술식 평가 CLI | single-file / 알고리즘 | 파서·우선순위·괄호 (`eval` 금지) |
| **L5** | `todo` 할 일 관리 CLI | multi-file / 서브커맨드 | 서브커맨드 + JSON 영속 + 종료코드 |
| **L6** | `csvstat` CSV 통계 CLI | multi-module / 데이터 | CSV 파싱 + 수치 집계 + 에러처리 |

> 채점은 모두 **숨은 독립 judge**가 한다 — L1은 import로 값 검증, 나머지는 subprocess **블랙박스**로
> stdout·종료코드·영속을 검증한다(도구 자기보고 불신).
>
> 두 축으로 분별력을 키웠다: **(1) 난이도 사다리**(단일파일→멀티파일→멀티모듈)와 **(2) 도메인 다양성**
> (계산·텍스트·상태저장·데이터처리). 한 도구가 어떤 *종류*의 과제에 강하고 약한지가 드러난다.

## 왜 여러 과제(난이도 + 도메인)인가

핵심 아이디어: **난이도와 도메인을 벌려가며 도구의 한계가 어디서 드러나는지**를 본다.

- **한 과제만 쓰면** 우연(쉬워서 둘 다 통과 / 어려워서 둘 다 실패)에 묻혀 차이가 안 보인다.
- **여러 과제로 벌리면** "어디까지는 둘 다 되고, 어디서부터·어떤 종류에서 갈라지는가"라는 **분기점**이 보인다.
- **난이도 축** — 각 단계는 앞 단계가 요구하지 않던 새 능력을 추가:
  - L1 → L2: 파일 1개 → **여러 파일 협력 + CLI 인터페이스 계약**
  - L2 → L3: 명령 실행 → **패키지 구조 + 프로세스 간 상태 영속**
- **도메인 축** — 같은 난이도라도 다른 *종류*의 문제를 섞어 약점을 노출:
  - L4 **계산**(알고리즘/파서) · L5 **상태저장 도구**(서브커맨드) · L6 **데이터 처리**(CSV/집계)
  - 예: "알고리즘은 잘하지만 데이터 에러처리는 약하다" 같은 차이가 드러난다.

> 그래서 결과를 보면 "능력의 ceiling"이 어디인지, 또 어떤 *종류*의 과제에서 약한지 읽힌다.

## 왜 이 구체적 과제들인가

- **L1 fib(피보나치)** — 군더더기 없는 단일 함수. "기본 코드를 정확히 쓰는가"의 바닥선.
- **L2 wordstat(단어 통계 CLI)** — 파싱·집계·정렬·표준출력의 *현실적 CLI* 최소형. 여러 파일 구조 +
  출력 형식 계약을 함께 본다.
- **L3 KV 스토어(키-값 서비스)** — 패키지 + **디스크 영속 + 프로세스 재시작 후 데이터 유지**. 실제
  서비스에 가까운 완성도.
- **L4 calc(산술식 평가)** — 단일 파일이지만 **`eval` 없이 파서**를 만들어 우선순위·괄호·단항 마이너스를
  정확히 처리해야 한다. *알고리즘 정확성*을 본다.
- **L5 todo(할 일 관리)** — 여러 **서브커맨드**(add/list/done/rm) + JSON 영속 + **안정적 id** + 성공/실패
  **종료코드**. 상태저장 CLI의 전형.
- **L6 csvstat(CSV 통계)** — CSV를 파싱해 수치 컬럼을 집계하고, 없는 파일·없는 컬럼·비수치 컬럼에
  **올바른 에러 종료코드**로 반응. *데이터 처리 + 견고성*을 본다.

여섯 과제 모두 **Python 표준 라이브러리만** 쓴다 — 외부 설치 없이 **어디서나 재현**되게 하기 위해서다.

## 공정성 장치 (모든 레벨 공통)

- **프롬프트는 한 곳에만**(`benchmark/tasks/<level>/PROMPT.md`), 두 도구에 **글자 그대로** 동일하게 준다.
- **judge는 도구에게 숨긴다** — 프롬프트는 테스트 내용을 모른다. 그래서 한 judge로 어느
  도구의 산출물이든 공정하게 판정하고, 도구가 테스트에 과적합할 수 없다.
- L2~L6는 **블랙박스**(subprocess)로 채점한다 — 내부 구현이 어떻든 *겉으로 약속한 동작*만 본다.
- 각 과제의 **결과물(파일 리스트)** 과 **기능·사용법**은 `benchmark/tasks/<level>/ABOUT.md` 에 정리돼 있다.

## 기대효과 (이 6종으로 무엇을 알게 되나)

| 무엇을 알고 싶나 | 어느 레벨이 답하나 |
|------------------|--------------------|
| **기본 코드**를 정확히 쓰는가 | L1 (둘 다 통과가 기대치) |
| **구조화된 CLI** + 출력 계약 | L2 |
| **멀티모듈 + 상태 영속**을 끝까지 완성하는가 | L3 |
| **알고리즘/파서**를 직접 구현하는가 | L4 |
| **서브커맨드 + 상태 + 종료코드**를 정확히 다루는가 | L5 |
| **데이터 파싱 + 에러 견고성** | L6 |
| 같은 능력이라도 **얼마나 빠른가** | 모든 레벨의 Time |
| 작업을 **어떤 식으로** 진행하는가(granularity) | 모든 레벨의 Steps(도구 내부 추세) |

> 요약하면: **난이도(L1→L3)와 도메인(L4 계산·L5 상태·L6 데이터)** 을 함께 벌려, 두 도구가 *어디서*·
> *어떤 종류*에서 갈라지는지를 본다. 그게 이 벤치마크의 핵심 관전 포인트다.
