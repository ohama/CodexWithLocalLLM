# 7. 결과 해석

수행이 끝나면 `benchmark/RESULTS.md` 에 표가 생긴다. 이 장은 **그 표를 어떻게 읽어야 하는가**를
다룬다 — 이 책에서 가장 중요한 부분이다. 숫자를 잘못 읽으면 틀린 결론을 내리기 쉽다.

## 표의 모양

각 행 = 한 칸(한 도구 × 한 레벨):

```
| Tool | Level | Success | Time | Steps (step_method) | Size (files / loc) |
```

## 4가지 지표 — 의미·측정·주의

### Success — 능력 (가장 중요)
- **의미:** 산출물이 실제로 과제를 푸는가.
- **측정:** 그 레벨의 독립 judge(`tasks/<level>/test.py`)를 **산출 폴더에 대해 다시 실행** →
  종료코드 0이면 PASS, 아니면 FAIL.
- **주의:** 도구가 "all tests passed" 라고 적어도 **무시**한다. 오직 judge 재실행 결과만 신뢰.

### Time — 속도
- **의미:** 그 과제를 푸는 데 걸린 실제 벽시계 시간(초).
- **측정:** 실행 시작~종료 시각 차이.
- **주의:** 같은 백엔드를 직렬로 쓰므로 도구 간 비교는 공정하다. 단, **실행마다 달라진다**(§비결정성).

### Steps — 작업량/과정
- **의미:** 도구가 목표까지 몇 "스텝"을 밟았나(작업 과정의 granularity).
- **측정 — 도구마다 단위가 다르다:**
  - codex = transcript의 `exec`(셸 실행) 블록 수
  - openhands = transcript의 "Number of agent messages" 값
- **⚠️ 절대 주의:** codex의 `2`와 openhands의 `4`는 **다른 종류의 사건**이다.
  **raw 숫자를 도구끼리 직접 비교하지 말 것.** 그래서 표에 항상 `step_method` 를 같이 적는다.
  같은 도구의 레벨 간 추세(L1<L2<L3)나, 같은 도구의 실행 간 비교에만 쓴다.

### Size — 산출 규모
- **의미:** 만들어 낸 파일 수와 코드 줄 수(LOC).
- **측정:** 산출 폴더의 파일/LOC 합. 단 `transcript.log`·`meta.json`·`__pycache__`·`.pyc` 제외(artifact).
- **주의:** 규모가 크다고 더 좋은 게 아니다. **반드시 Success와 함께** 본다(0 files면 보통 실패).

## 표 읽는 순서 (추천)

1. **Success 열 먼저** — 누가 어떤 레벨까지 통과했나(능력의 1차 지표).
2. **레벨별로 두 도구 비교** — 같은 레벨에서 codex vs openhands의 성공·시간·규모.
3. **Time** — 같은 PASS라도 누가 더 빨랐나.
4. **Steps는 `step_method` 확인 후** 같은 도구 안에서만(또는 추세로만).
5. **`0 files` + `FAIL`** 조합이 보이면 → §이상치 진단으로.

## 올바른 결론 vs 위험한 결론

- ✅ "openhands가 L3를 통과했고 codex는 실패했다(이 실행에서)" — Success 기반, 타당.
- ✅ "codex가 L1/L2에서 더 빨랐다" — Time 기반, 타당.
- ❌ "codex가 steps가 적으니 더 효율적이다" — **단위가 달라 비교 불가**. `step_method` 무시한 오독.
- ❌ "openhands가 loc가 많으니 더 낫다" — 규모 ≠ 품질. Success와 분리하면 의미 없음.

## 왜 결과가 매번 다른가 (비결정성)

LLM은 같은 프롬프트라도 실행마다 다른 출력을 낸다. 따라서:
- **한 번의 매트릭스는 "스냅샷"** 이다 — 절대적 순위표가 아니라 그 실행에서 일어난 일이다.
- 특히 경계선 과제(L3)는 **같은 도구가 어떤 날은 통과, 어떤 날은 실패**할 수 있다.
- 더 견고한 결론을 원하면 같은 칸을 여러 번 돌려 통과율/시간 분포로 봐야 한다(현재 1회 측정).

## 이상치 진단 — "실패"가 진짜 실패인가?

`FAIL` 이라고 다 같은 실패가 아니다. transcript와 산출 폴더를 보고 구분한다:

| 패턴 | 가능한 원인 | 확인법 |
|------|-------------|--------|
| FAIL + **0 files**, 매우 짧은 시간 | 도구가 **일찍 잘림/포기**(진짜 실패) | transcript 끝이 중간에 끊겼나 |
| FAIL + **0 files**, 정상 시간 | 산출물이 **다른 위치로 샘**(격리/하니스 문제) | 엉뚱한 폴더에 파일이 생겼나 |
| FAIL + 파일 있음 | 코드는 썼으나 **계약 위반**(진짜 실패) | judge를 직접 돌려 어느 검사가 깨지나 |

> **`0 files` FAIL은 항상 한 번 의심하라.** 이 벤치마크 v1에서 openhands L2/L3가 처음엔
> "0 files FAIL"로 찍혔지만, 진단해 보니 **격리 누출**(출력이 정규 폴더로 샘)이었다 → 러너 수정 후
> 재측정하니 PASS. 반면 codex L3는 14초 만에 잘린 **진짜 실패**라 그대로 뒀다. 이런 구분을 리포트의
> "honesty note"에 남긴다.

judge를 직접 돌려 확인:
```sh
python3 benchmark/tasks/l3-kvstore/test.py benchmark/.runs/<해당-run-폴더>
echo "exit=$?"   # 0=PASS, nonzero=FAIL
```
