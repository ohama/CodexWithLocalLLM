Codex vs OpenHands 벤치마크 — 따라하며 배우는 튜토리얼

이 책은 로컬 LLM(qwen-122b) 위에서 두 코딩 에이전트 — Codex CLI 와 OpenHands — 를 같은 조건으로 비교 검증하는 과정을, 초보 개발자가 위에서 아래로 따라가며 직접 수행하고 그 결과를 해석할 수 있게 정리한 것이다.

이 책이 답하는 질문

“내 로컬 모델로 codex와 openhands 중 어느 쪽이, 어떤 작업에서, 얼마나 잘/빠르게 하는가? 그리고 그 결과를 어떻게 믿고 읽어야 하는가?”

큰 그림

두 도구(codex, openhands)
   │   같은 모델(qwen-122b) · 같은 과제 · 같은 조건
   ▼
같은 3단계 과제를 시킴 (쉬움→어려움)
   │
   ▼
성공 여부·시간·과정·규모를 자동 측정
   │
   ▼
표로 비교하고 → 올바르게 해석

이 책의 흐름 (수행 → 해석)

  1. 개요 — 무엇을·왜 비교하나, 공정성의 3원칙
  2. 과제 3종 — 왜 이 3개(fib/wordstat/KV)를 골랐나 + 기대효과
  3. 사전 준비 — 백엔드(qwen-122b)가 떠 있는지 확인
  4. Codex 연결 — codex를 로컬 모델에 붙이기
  5. OpenHands 연결 — openhands를 같은 모델에 붙이기
  6. 수행 — 한 셀 시험 → 전체 매트릭스 실행
  7. 해석 — 4가지 지표의 의미와 표 읽는 법, 흔한 오독
  8. 결과 요약과 결론 — 실제 결과 + 무엇을 말할 수 있나
  9. 막혔을 때 — 트러블슈팅

비교 대상(도구)은 codex 와 openhands 둘뿐이고, 두 도구에 똑같이 시키는 과제가 3종(L1/L2/L3) 이다. (다른 도구는 이 책의 범위 밖.)

준비됐으면 1. 무엇을 왜 비교하나부터 시작하자.

1. 무엇을 왜 비교하나

비교의 목적

같은 코딩 과제를, 같은 모델로, 같은 조건에서 두 도구에 시켜서 “도구의 차이“만 드러낸다. 모델을 통일하기 때문에 결과 차이는 모델 성능이 아니라 각 도구가 일하는 방식에서 온다.

  • Codex CLI — OpenAI의 터미널 코딩 에이전트(여기선 로컬 qwen-122b에 연결해서 사용).
  • OpenHands — 오픈소스 코딩 에이전트(action/observation 루프 + 컨텍스트 압축).

둘 다 이 시스템에선 동일한 로컬 백엔드(qwen-122b @ LiteLLM :4000) 를 쓴다.

공정성의 3원칙

이 벤치마크가 신뢰할 만한 이유는 세 가지 원칙 때문이다. 책 전체가 이 원칙 위에 서 있다.

① 동일 조건

  • 같은 프롬프트(글자 그대로), 같은 모델(qwen-122b), 격리된 작업 폴더, 직렬 실행.
  • 두 도구가 같은 출발선에 선다.

② 공정한 채점 (자기보고 불신)

  • 성공 여부는 도구가 “다 됐어요“라고 말한 걸 믿지 않는다.
  • 대신 독립 judge 가 도구의 산출물을 다시 검사해 통과/실패를 정한다.
  • 즉 “테스트 다 통과했다“는 도구의 transcript 문구는 무시하고, 우리 judge의 종료코드만 본다.

③ 정직한 기록

  • 실패·재측정·이상치를 숨기지 않고 결과 리포트에 그대로 남긴다.
  • 예: 어떤 도구가 “성공“이라 우겨도 judge가 실패로 판정하면 실패로 기록한다.

왜 같은 모델(qwen-122b)인가

만약 codex는 큰 모델, openhands는 작은 모델로 비교한다면 — 결과 차이가 도구 때문인지 모델 때문인지 알 수 없다. 모델을 하나로 고정해야 비교가 “도구 대 도구“가 된다.

이 책에서 측정하는 것은 도구의 작업 방식 차이(같은 모델을 어떻게 굴리는가)이지, 모델 자체의 성능 순위가 아니다.

다음

무엇을 비교하는지 알았으니, 이제 2. 과제 3종 — 선정 이유와 기대효과로 넘어가자.

2. 벤치마크 과제 3종 — 선정 이유와 기대효과

비교 대상은 도구 2개(codex · openhands)이고, 그 둘에게 똑같이 시키는 과제가 3종이다. 왜 하필 이 3개인가? 그리고 각 과제에서 무엇을 보려고 하는가(기대효과)?

한눈에

레벨과제복잡도 차원채점 방식이 레벨이 가리는(분별하는) 것
L1fib(n) 단일 파일single-filejudge가 fib 를 import 해 값 검증기본기 — 함수 하나를 정확히 쓰는가
L2wordstat 멀티파일 CLImulti-file CLIjudge가 subprocess 로 실행, stdout/종료코드 검증협력 — 여러 파일로 나누고 표준출력 계약을 지키는가
L3KV 스토어 멀티모듈 서비스multi-module servicejudge가 별도 프로세스로 CLI 호출, 영속·종료코드 계약 검증완성력 — 패키지 + 상태 영속까지 끝까지 만드는가

왜 “3단계 ladder“인가

핵심 아이디어: 난이도를 한 칸씩 올려가며 도구의 한계가 어디서 드러나는지를 본다.

  • 한 과제만 쓰면 우연(쉬워서 둘 다 통과 / 어려워서 둘 다 실패)에 묻혀 차이가 안 보인다.
  • 3단계로 벌리면 “어디까지는 둘 다 되고, 어디서부터 갈라지는가“라는 분기점이 보인다.
  • 각 단계는 앞 단계가 요구하지 않던 새 능력을 추가로 요구하도록 설계했다:
    • L1 → L2: 파일 1개 → 여러 파일 협력 + CLI 인터페이스 계약
    • L2 → L3: 명령 실행 → 패키지 구조 + 프로세스 간 상태 영속

그래서 결과를 보면 “능력의 ceiling“이 어디인지 읽힌다. 둘 다 L1·L2는 통과하는데 L3에서 갈린다면, 그 도구의 한계는 대형·다구성 과제를 끝까지 완성하는 지점에 있다는 뜻이다.

왜 이 구체적 과제(fib / wordstat / KV)인가

  • fib(피보나치) — 군더더기 없는 단일 함수. “도구가 기본적인 코드를 정확히 쓰는가“의 바닥선.
  • wordstat(단어 통계 CLI) — 입력 파싱·집계·정렬·표준출력이라는 현실적인 CLI 의 최소형. 여러 파일로 나누는 구조와 출력 형식 계약을 동시에 본다.
  • KV 스토어(키-값 서비스) — 패키지(모듈 여러 개) + 디스크 영속 + 프로세스 재시작 후에도 데이터 유지 라는, 실제 서비스에 가까운 요구. “장난감“을 넘는 완성도를 본다.

또한 세 과제 모두 Python 표준 라이브러리만 쓴다 — 외부 설치 없이 어디서나 재현되게 하기 위해서다.

공정성 장치 (모든 레벨 공통)

  • 프롬프트는 한 곳에만(benchmark/tasks/<level>/PROMPT.md), 두 도구에 글자 그대로 동일하게 준다.
  • judge는 도구에게 숨긴다 — 프롬프트는 테스트 내용을 모른다. 그래서 한 judge로 어느 도구의 산출물이든 공정하게 판정하고, 도구가 테스트에 과적합할 수 없다.
  • L2·L3는 블랙박스(subprocess)로 채점한다 — 내부 구현이 어떻든 겉으로 약속한 동작만 본다.

기대효과 (이 3종으로 무엇을 알게 되나)

무엇을 알고 싶나어느 레벨이 답하나
두 도구가 기본 코드를 정확히 쓰는가L1 (둘 다 통과가 기대치)
구조화된 CLI를 만들고 출력 계약을 지키는가L2
복잡한 멀티모듈 서비스를 끝까지 완성하는가 (도구의 ceiling)L3 (여기서 갈릴 가능성↑)
같은 능력이라도 얼마나 빠른가모든 레벨의 Time
작업을 어떤 식으로 진행하는가(granularity)모든 레벨의 Steps(도구 내부 추세)

요약하면: L1=바닥선, L2=현실적 중간, L3=ceiling 시험. 두 도구가 이 ladder의 어디서 갈라지는지가 이 벤치마크의 핵심 관전 포인트다.

다음

과제 설계를 이해했으니, 3. 사전 준비에서 백엔드를 확인하고 실제로 돌려 보자. (실제 결과에서 이 ladder가 어떻게 갈렸는지는 8. 결과 요약과 결론에서 본다.)

3. 사전 준비 — 백엔드 확인

두 도구 모두 로컬 LiteLLM 게이트웨이(:4000) 를 통해 같은 qwen-122b를 쓴다. 비교를 시작하기 전에 이 백엔드가 살아 있는지부터 확인한다.

환경변수 (게이트웨이 키)

LiteLLM은 아무 키나 받는다. 셸에 더미 키를 넣는다:

export LITELLM_API_KEY=dummy
  • 효과: codex/openhands/채점 스크립트가 게이트웨이에 붙을 때 쓰는 키. 값은 아무거나 OK.

게이트웨이가 떠 있나 (주 점검)

curl -sf localhost:4000/v1/models -H "Authorization: Bearer dummy" >/dev/null && echo "gateway OK" || echo "gateway DOWN"
  • 효과: gateway OK 면 백엔드 준비 완료 → 다음 장으로.
  • gateway DOWN 이면 백엔드 스택을 먼저 띄워야 한다(아래).

백엔드 스택의 구성 (DOWN일 때)

이 시스템의 로컬 LLM 스택은 3개 층이다:

도구 ──▶ LiteLLM :4000 (게이트웨이) ──▶ role-shim :8011 ──▶ mlx_lm.server :8001 (qwen-122b)

각 층이 떠 있는지 확인:

lsof -ti tcp:4000 >/dev/null && echo ":4000 LiteLLM UP"   || echo ":4000 DOWN"
lsof -ti tcp:8011 >/dev/null && echo ":8011 role-shim UP" || echo ":8011 DOWN"
lsof -ti tcp:8001 >/dev/null && echo ":8001 mlx UP"       || echo ":8001 DOWN"
  • 이 시스템에선 이 셋이 launchd 서비스로 항상 떠 있다: com.ohama.litellm, com.ohama.role-shim, com.ohama.qwen122b.
  • 내려갔다면 재시작: 
    launchctl kickstart -k gui/$(id -u)/com.ohama.litellm
  • 스택을 처음부터 구축하는 방법(설치·shim·브릿지)은 별도 문서 documentation/howto/connect-codex-to-local-qwen122b.md 에 있다.

필요한 도구가 설치돼 있나

codex --version       # 예: codex-cli 0.142.0
openhands --version   # 예: OpenHands CLI 1.16.0
  • 둘 다 버전이 나오면 설치 완료. 안 나오면 다음 두 장(Codex/OpenHands 연결)에서 설치한다.

체크리스트

  • export LITELLM_API_KEY=dummy 했다
  • gateway OK 가 나온다
  • codex --version / openhands --version 둘 다 버전이 나온다

다 됐으면 4. Codex 연결로.

4. Codex 를 qwen-122b 에 연결

이미 codex 가 로컬 qwen 으로 동작하면 이 장은 건너뛴다.

연결됐는지 확인

grep -E '^[[:space:]]*model[[:space:]]*=' ~/.codex/config.toml
# → model = "qwen-122b-codex" 가 보이면 연결됨

연결 원리 (왜 이게 필요한가)

최신 Codex CLI를 로컬 백엔드에 붙일 때 두 개의 호환성 벽이 있다:

  1. Codex가 chat API를 버리고 Responses API만 쓴다. → LiteLLM이 모델 id에 openai/chat_completions/ prefix를 붙여 responses→chat 으로 번역(브릿지).
  2. 백엔드 mlx_lm.server 가 Codex의 developer role을 거부한다. → 그 사이에 role-shim(developer→system)을 끼운다.
Codex ──Responses──▶ LiteLLM :4000 (qwen-122b-codex)
        ──브릿지──▶ role-shim :8011 ──▶ mlx_lm.server :8001 (qwen-122b)

설정 요지

~/.codex/config.toml:

model = "qwen-122b-codex"
model_provider = "litellm"

[model_providers.litellm]
name = "LiteLLM (local Qwen)"
base_url = "http://localhost:4000/v1"
env_key = "LITELLM_API_KEY"
wire_api = "responses"

전체 설치 과정(브릿지 모델 등록 + role-shim launchd 서비스 + Codex CLI 설치)은 documentation/howto/connect-codex-to-local-qwen122b.md 에 단계별로 있다. 이 장은 그 요지다.

동작 확인 (선택)

LITELLM_API_KEY=dummy codex exec --skip-git-repo-check --sandbox read-only \
  "Reply with exactly: OK" < /dev/null
  • OK 가 나오면 codex→qwen-122b 경로가 살아 있다.

⚠️ 백그라운드/파이프에서 codex exec 를 돌릴 땐 항상 < /dev/null 을 붙인다. 안 그러면 stdin 입력을 기다리며 멈춘다(hang). 벤치마크 러너는 이걸 자동으로 처리한다.

다음

codex가 연결됐으면 5. OpenHands 연결로.

5. OpenHands 를 qwen-122b 에 연결

이미 openhands 가 로컬 qwen 으로 동작하면 건너뛴다.

연결됐는지 확인

python3 -c "import json,os;print(json.load(open(os.path.expanduser('~/.openhands/agent_settings.json')))['llm']['model'])"
# → openai/qwen-122b 가 나오면 연결됨

설치

OpenHands CLI는 uv tool 로 설치되어 ~/.local/bin/openhands 를 제공한다:

openhands --version          # 이미 있으면 버전 출력 (예: OpenHands CLI 1.16.0)
# 없으면:
uv tool install openhands    # 또는 공식 설치 안내를 따른다

모델을 qwen-122b 로 지정

~/.openhands/agent_settings.json 의 llm(그리고 condenser.llm)을 로컬 게이트웨이로 맞춘다:

키값
llm.modelopenai/qwen-122b
llm.base_urlhttp://localhost:4000/v1
llm.api_keydummy

codex와 같은 qwen-122b 로 맞추는 게 핵심이다 — 그래야 “도구 차이“만 비교된다(모델 변수 제거). OpenHands는 내부 컨텍스트 압축기(condenser)도 LLM을 쓰므로 condenser.llm 도 같은 모델로 둔다.

설정 후 위 확인 명령으로 openai/qwen-122b 가 나오는지 다시 본다.

OpenHands 의 한 가지 함정 (중요)

OpenHands 를 손으로 직접 호출하지 말 것. 특히 프롬프트를 --file 로 주면, OpenHands가 그 파일이 있는 디렉터리를 작업 위치로 잡아 솔루션을 엉뚱한 폴더에 쓰는 격리 누출이 생긴다(이 벤치마크 v1에서 실제로 발견·수정한 버그).

→ 항상 벤치마크 러너(run.sh / run-matrix.sh)를 통해서만 실행한다. 러너는 프롬프트를 인라인 (--task)으로 주고 작업 폴더를 OPENHANDS_WORK_DIR 로 못박아 이 문제를 막는다.

체크리스트

  • openhands --version 이 나온다
  • 확인 명령이 openai/qwen-122b 를 출력한다

이제 두 도구 모두 같은 모델에 연결됐다 → 6. 벤치마크 수행으로.

6. 벤치마크 수행

이제 실제로 두 도구에 과제를 시킨다. 모든 실행은 저장소 루트(/Users/ohama/projs/codex)에서 한다.

export LITELLM_API_KEY=dummy   # 아직 안 했으면

과제 3단계 (복잡도 ladder)

레벨과제무엇을 보나
L1fib(n) 단일 파일함수 한 개를 정확히
L2wordstat 멀티파일 CLI여러 파일 협력 + 표준출력 계약
L3KV 스토어 멀티모듈 서비스패키지 구조 + 프로세스 간 영속성

각 과제의 프롬프트는 benchmark/tasks/<level>/PROMPT.md 한 곳에만 있고, 두 도구에 글자 그대로 동일하게 주어진다. judge benchmark/tasks/<level>/test.py 는 도구가 못 보는 숨은 judge다.

① 한 셀 먼저 시험

전체를 돌리기 전에, 한 칸(한 도구 × 한 레벨)을 먼저 돌려 흐름을 본다:

bash benchmark/run.sh codex l1
  • 효과: codex에게 L1 과제를 시키고 → 격리 폴더 benchmark/.runs/codex-l1-fib-<시각>/ 에 솔루션 + transcript.log(대화 기록) + meta.json(4지표) 을 남긴다 → 채점까지 자동.

openhands도 똑같이:

bash benchmark/run.sh openhands l1

한 셀은 보통 30초~수분. L3가 가장 오래 걸린다.

② 전체 매트릭스 (6칸)

두 도구 × 세 레벨 = 6칸을 순서대로(직렬) 돌린다:

bash benchmark/run-matrix.sh
  • 효과: codex L1→L2→L3, openhands L1→L2→L3 순으로 한 번에 하나씩 실행하고, 6개 결과를 하나의 results.json 으로 모은다. 전체 ~10–15분.
  • ⚠️ 절대 백그라운드(&)로 돌리지 말 것. 로컬 백엔드가 하나라 동시에 두 개를 돌리면 서로 경합한다. 러너가 직렬을 강제하지만, 사람이 백그라운드로 띄우면 깨질 수 있다.

③ 리포트 생성

python3 benchmark/report.py
  • 효과: results.json 을 읽어 사람이 읽는 리포트 benchmark/RESULTS.md 를 만든다(표 + 각 실행 transcript 발췌 + 레벨별 차이 + honesty note).
  • benchmark/.runs/ 는 git에 올라가지 않으므로 RESULTS.md 가 그 실행의 유일한 영구 기록이다.

④ 결과 열기

open benchmark/RESULTS.md     # 또는 에디터로 연다

여기까지가 “수행“이다. 이 표를 어떻게 읽어야 하는지가 다음 장의 핵심이다 → 7. 결과 해석.

한 줄 요약

export LITELLM_API_KEY=dummy
bash benchmark/run-matrix.sh     # 6칸 실행 (직렬, ~10-15분)
python3 benchmark/report.py      # → benchmark/RESULTS.md

7. 결과 해석

수행이 끝나면 benchmark/RESULTS.md 에 표가 생긴다. 이 장은 그 표를 어떻게 읽어야 하는가를 다룬다 — 이 책에서 가장 중요한 부분이다. 숫자를 잘못 읽으면 틀린 결론을 내리기 쉽다.

표의 모양

각 행 = 한 칸(한 도구 × 한 레벨):

| Tool | Level | Success | Time | Steps (step_method) | Size (files / loc) |

4가지 지표 — 의미·측정·주의

Success — 능력 (가장 중요)

  • 의미: 산출물이 실제로 과제를 푸는가.
  • 측정: 그 레벨의 독립 judge(tasks/<level>/test.py)를 산출 폴더에 대해 다시 실행 → 종료코드 0이면 PASS, 아니면 FAIL.
  • 주의: 도구가 “all tests passed” 라고 적어도 무시한다. 오직 judge 재실행 결과만 신뢰.

Time — 속도

  • 의미: 그 과제를 푸는 데 걸린 실제 벽시계 시간(초).
  • 측정: 실행 시작~종료 시각 차이.
  • 주의: 같은 백엔드를 직렬로 쓰므로 도구 간 비교는 공정하다. 단, 실행마다 달라진다(§비결정성).

Steps — 작업량/과정

  • 의미: 도구가 목표까지 몇 “스텝“을 밟았나(작업 과정의 granularity).
  • 측정 — 도구마다 단위가 다르다:
    • codex = transcript의 exec(셸 실행) 블록 수
    • openhands = transcript의 “Number of agent messages” 값
  • ⚠️ 절대 주의: codex의 2와 openhands의 4는 다른 종류의 사건이다. raw 숫자를 도구끼리 직접 비교하지 말 것. 그래서 표에 항상 step_method 를 같이 적는다. 같은 도구의 레벨 간 추세(L1<L2<L3)나, 같은 도구의 실행 간 비교에만 쓴다.

Size — 산출 규모

  • 의미: 만들어 낸 파일 수와 코드 줄 수(LOC).
  • 측정: 산출 폴더의 파일/LOC 합. 단 transcript.log·meta.json·__pycache__·.pyc 제외(artifact).
  • 주의: 규모가 크다고 더 좋은 게 아니다. 반드시 Success와 함께 본다(0 files면 보통 실패).

표 읽는 순서 (추천)

  1. Success 열 먼저 — 누가 어떤 레벨까지 통과했나(능력의 1차 지표).
  2. 레벨별로 두 도구 비교 — 같은 레벨에서 codex vs openhands의 성공·시간·규모.
  3. Time — 같은 PASS라도 누가 더 빨랐나.
  4. Steps는 step_method 확인 후 같은 도구 안에서만(또는 추세로만).
  5. 0 files + FAIL 조합이 보이면 → §이상치 진단으로.

올바른 결론 vs 위험한 결론

  • ✅ “openhands가 L3를 통과했고 codex는 실패했다(이 실행에서)” — Success 기반, 타당.
  • ✅ “codex가 L1/L2에서 더 빨랐다” — Time 기반, 타당.
  • ❌ “codex가 steps가 적으니 더 효율적이다” — 단위가 달라 비교 불가. step_method 무시한 오독.
  • ❌ “openhands가 loc가 많으니 더 낫다” — 규모 ≠ 품질. Success와 분리하면 의미 없음.

왜 결과가 매번 다른가 (비결정성)

LLM은 같은 프롬프트라도 실행마다 다른 출력을 낸다. 따라서:

  • 한 번의 매트릭스는 “스냅샷” 이다 — 절대적 순위표가 아니라 그 실행에서 일어난 일이다.
  • 특히 경계선 과제(L3)는 같은 도구가 어떤 날은 통과, 어떤 날은 실패할 수 있다.
  • 더 견고한 결론을 원하면 같은 칸을 여러 번 돌려 통과율/시간 분포로 봐야 한다(현재 1회 측정).

이상치 진단 — “실패“가 진짜 실패인가?

FAIL 이라고 다 같은 실패가 아니다. transcript와 산출 폴더를 보고 구분한다:

패턴가능한 원인확인법
FAIL + 0 files, 매우 짧은 시간도구가 일찍 잘림/포기(진짜 실패)transcript 끝이 중간에 끊겼나
FAIL + 0 files, 정상 시간산출물이 다른 위치로 샘(격리/하니스 문제)엉뚱한 폴더에 파일이 생겼나
FAIL + 파일 있음코드는 썼으나 계약 위반(진짜 실패)judge를 직접 돌려 어느 검사가 깨지나

0 files FAIL은 항상 한 번 의심하라. 이 벤치마크 v1에서 openhands L2/L3가 처음엔 “0 files FAIL“로 찍혔지만, 진단해 보니 격리 누출(출력이 정규 폴더로 샘)이었다 → 러너 수정 후 재측정하니 PASS. 반면 codex L3는 14초 만에 잘린 진짜 실패라 그대로 뒀다. 이런 구분을 리포트의 “honesty note“에 남긴다.

judge를 직접 돌려 확인:

python3 benchmark/tasks/l3-kvstore/test.py benchmark/.runs/<해당-run-폴더>
echo "exit=$?"   # 0=PASS, nonzero=FAIL

8. 결과 요약과 결론

앞 장의 해석법을 실제 측정값(v1.0, qwen-122b)에 적용해 보자. 이건 한 번의 스냅샷이며, 당신이 직접 돌리면 숫자는 달라질 수 있다.

⏩ 결과 한눈 요약 (TL;DR)

통과 레벨속도(L1·L2)ceiling(L3)한 줄
codexL1·L2 (L3 실패)더 빠름 (1.5~2배)❌ truncate로 빈 패키지쉬운~중간 과제를 빠르게 끝냄, 대형은 끝까지 못 감(이 실행)
openhandsL1·L2·L3 전부더 느림✅ 완성(9파일/362줄)느리지만 복잡한 과제까지 완성

과제 ladder가 어디서 갈렸나: L1·L2는 무승부(둘 다 통과), L3에서 분기 — 가장 복잡한 과제에서 codex의 ceiling이 드러났다. (왜 이 ladder로 분기를 본 건지는 2장 참고.)

기대효과 vs 실제(이 스냅샷):

  • L1 “둘 다 통과 기대” → ✅ 맞음.
  • L2 “구조화 CLI” → ✅ 둘 다 통과.
  • L3 “ceiling 시험, 갈릴 가능성↑” → ✅ 예상대로 갈림(codex 실패 / openhands 통과).

⚠️ 단, 이건 1회 측정이다. “codex가 L3를 못 한다“가 아니라 “이 실행에서 truncate됐다“이다. 확정하려면 반복 측정으로 통과율을 봐야 한다(아래 결론 참고).

측정 결과 (예시)

ToolLevelSuccessTimeSteps (step_method)Size
codexL1 fib✅ PASS26s2 (codex: exec blocks)1f / 31loc
codexL2 wordstat✅ PASS98s10 (codex: exec blocks)4f / 148loc
codexL3 kvstore❌ FAIL14s2 (codex: exec blocks)0f / 0loc

해석 (§7의 순서대로 읽기)

1) Success 먼저:

  • codex: L1·L2 통과, L3 실패.
  • openhands: L1·L2·L3 전부 통과.

2) 레벨별 비교:

  • L1·L2 — 둘 다 성공. 능력은 대등(이 레벨에선).
  • L3(가장 어려움) — 여기서 갈렸다. openhands는 완성, codex는 실패.

3) Time:

  • codex가 L1(26 vs 49s)·L2(98 vs 145s)에서 더 빠르다(약 1.5~2배).
  • 즉 “쉬운~중간 과제는 codex가 빠르게 끝낸다”.

4) Steps (step_method 확인 후, 도구 내부만):

  • codex L1→L2: 2→10 (어려워질수록 작업량↑) — 추세로만 읽는다.
  • openhands L1→L2→L3: 4→16→15.
  • codex의 10과 openhands의 16을 직접 비교하지 않는다(단위가 다름).

5) 이상치 — codex L3 FAIL 진단:

  • 0 files + 14초(매우 짧음) → §7 표의 “일찍 잘림(진짜 실패)” 패턴.
  • transcript를 보면 mkdir kvstore 만 하고 끊겼다 → codex가 L3에서 truncate되어 빈 패키지만 남기고 멈춤. 격리 누출이 아니라 진짜 실패다.

이 스냅샷이 말하는 것 / 말하지 못하는 것

말할 수 있는 것:

  • 이 실행에서 codex는 빠르지만 가장 복잡한 과제(L3)를 완성하지 못했고, openhands는 더 느리지만 세 레벨을 모두 완성했다.
  • 작업 규모도 openhands가 L3에서 더 컸다(9파일/362줄 vs codex 0).

말하지 못하는 것:

  • “openhands가 항상 codex보다 낫다” — ❌. 한 번의 스냅샷이고 L1·L2는 codex가 더 빨랐다.
  • “codex는 L3를 못 한다” — ❌. truncation은 실행마다 다를 수 있다. 여러 번 돌려 통과율로 봐야 한다.
  • steps로 효율 비교 — ❌. 단위가 다르다.

결론 (균형 잡힌 읽기)

속도가 중요하고 과제가 단순~중간이면 codex가 유리할 수 있고, 복잡한 멀티모듈 과제를 끝까지 완성하는 안정성이 중요하면 openhands가 유리할 수 있다 — 이 한 번의 측정 기준으로는. 확정하려면 같은 매트릭스를 여러 번 돌려 통과율·시간 분포를 봐야 한다.

당신의 결과로 직접 이 해석을 해보는 것이 이 책의 목적이다. 막히면 → 9. 막혔을 때.
