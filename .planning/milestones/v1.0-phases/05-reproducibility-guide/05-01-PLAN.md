<objective>
Write the reproducibility guide that ties the existing, already-working benchmark harness
together so a human can reproduce the entire measurement from the documentation alone. This is
the FINAL plan of the milestone and closes REPRO-01..REPRO-03.

The deliverable is one committed document, `benchmark/REPRODUCE.md`, with three things:
1. A preconditions section whose checks are real commands with known-good expected output (REPRO-02).
2. A top-to-bottom run guide where every command is paired with its effect (REPRO-01).
3. A single "re-run from scratch" section giving the exact command sequence (REPRO-03).

Purpose: The project's core value is reproducibility, not the numbers. A reader must be able to
follow the doc on a clean checkout and either reproduce the matrix or know exactly what to expect.
Output: `benchmark/REPRODUCE.md` (new) + a one-line pointer added to `benchmark/README.md`.

HARD SCOPE: documentation only. No new harness code, no re-running the matrix, no new metrics.
The guide must drive runs through `bash benchmark/run.sh` / `bash benchmark/run-matrix.sh` — it must
NEVER tell the reader to hand-invoke openhands with `--file` (that isolation leak was fixed in
Phase 4; run.sh uses inline `--task`).
</objective>

Backend chain (all on localhost): LiteLLM gateway :4000 -> role-shim :8011 -> mlx_lm.server :8001

Model config (values, not secrets):
- codex: ~/.codex/config.toml has `model = "qwen-122b-codex"`, provider litellm, base_url http://localhost:4000/v1
- openhands: ~/.openhands/agent_settings.json -> llm.model = "openai/qwen-122b", llm.base_url = http://localhost:4000/v1
- Both report the model string differently but it is the same qwen-122b backend.

Env: LITELLM_API_KEY=dummy (run.sh defaults it to "dummy" if unset; document setting it explicitly).

Tools (confirmed on PATH): `codex --version`, `openhands --version`. `jq` is available but NOT required
(the harness uses python3/grep for JSON/TOML — keep the guide dependency-free, stdlib only).

Serial constraint: single mlx backend -> exactly one cell at a time. run.sh holds a mkdir-mutex
(.runs/.lock, exit 3 if held); run-matrix.sh calls run.sh serially and must NOT be backgrounded.

Real timing ballpark from the committed matrix (benchmark/RESULTS.md, matrix-20260626-042701):
- L1 (fib):      codex 26s  / openhands 49s     (~30-50s)
- L2 (wordstat): codex 98s  / openhands 145s    (~100-145s)
- L3 (kvstore):  codex 14s FAIL (genuine truncation: only ran `mkdir kvstore`, 0f/0loc) / openhands 147s PASS  (~145s+)
- Full 6-cell matrix, serial: ~10-15 min total.

Artifacts a run produces:
- run.sh -> benchmark/.runs/<tool>-<level>-<stamp>/ containing transcript.log + meta.json (4 metrics
  via score.py auto-hook: passed, duration_seconds, steps[step_method], files/loc).
- run-matrix.sh -> benchmark/.runs/matrix-<stamp>/ containing results.json + manifest.txt + *.console.log.
- report.py -> benchmark/RESULTS.md (auto-discovers latest matrix-*/results.json; idempotent; zero LLM time).

Durability: benchmark/.runs/ is gitignored (benchmark/.gitignore = `.runs/`). Therefore RESULTS.md is
the ONLY durable record and must be committed after re-running report.py.
